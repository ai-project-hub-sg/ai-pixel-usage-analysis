# AI Pixel Usage Analysis 设计规格

- 日期：2026-07-11
- 状态：已完成方案评审，等待书面规格复核
- 目标版本：v1
- 开发分支：`codex/complete-project`

## 1. 目标

构建一个可部署在 Linux 服务器上的 AI Pixel 用量分析服务。服务从多个上游账户定时采集请求记录和余额流水，持久化到本地 SQLite，提供全账户汇总及单账户分析，并通过需要登录的 Web 页面展示结果。

生产环境由 Nginx 提供公网 HTTPS 和反向代理。Go 服务只监听配置指定的本机地址与端口。前端构建产物嵌入 Go 二进制，运行时不依赖 Node.js、Python、外部数据库或其他应用运行时。

## 2. v1 范围

### 2.1 必须交付

- 多账户配置、独立认证、独立同步和全账户汇总。
- 按优先级和声明顺序进行上游节点选择与故障切换。
- 首次启动同步上月第一天至当前时间的请求记录和余额流水。
- 后续每分钟增量同步，支持幂等写入、回看窗口和断点续传。
- 默认小时聚合，并支持日粒度查看。
- 当前小时与昨天同小时、上周同小时分别对比。
- 余额入账与出账分开统计，并计算净变化。
- 流水原始类型、业务分类、方向和备注内容作为独立分析维度。
- 分析页面登录保护，服务端会话 Cookie 有效期为 24 小时；网页登录凭据仅由 `.env` 提供。
- 启动时若 `.env` 缺少网页登录用户名或密码，使用密码学安全随机数生成缺失项并安全回写。
- Linux 单二进制构建及 Nginx 部署文档。
- PRD、UI、TRD、项目结构、开发规范、验收方案和打包部署文档。

### 2.2 不在 v1 范围

- 在 Web 页面中编辑上游账户或凭据。
- 多个分析系统管理员、角色或细粒度权限。
- 外部 AI 服务参与备注分类。
- 分布式部署、外部数据库或多实例任务协调。
- 自动删除历史数据。

## 3. 技术方案与部署架构

采用 Go 单体服务、内嵌 React/TypeScript 前端和 SQLite：

1. React/TypeScript 负责登录页和分析界面，构建为静态资源。
2. Go 使用 `go:embed` 将前端静态资源打入最终二进制。
3. Go 提供分析 API、认证会话、定时采集、上游客户端、聚合查询和静态资源服务。
4. SQLite 使用纯 Go 驱动；Linux 构建使用 `CGO_ENABLED=0`。
5. Nginx 是唯一公网入口，负责 HTTPS、证书、安全响应头和反向代理。
6. Go 默认监听 `127.0.0.1`，监听端口由 `config.toml` 配置。

请求链路如下：

`浏览器 -> Nginx HTTPS -> Go HTTP -> SQLite / 上游 HTTPS`

## 4. 配置设计

非敏感配置保存在 `config.toml`；上游密码可由环境变量或 `.env` 提供，分析管理员明文凭据只保存在项目 `.env` 中。`.env` 必须保持在 Git 忽略列表中。`config.toml` 不保存网页登录用户名、密码、密码哈希，也不保存这些字段所使用的环境变量名。

配置结构示例：

```toml
[server]
host = "127.0.0.1"
port = 8080
public_url = "https://usage.example.com"
secure_cookie = true

[analysis]
timezone = "Asia/Shanghai"
sync_interval = "1m"
sync_overlap = "5m"
preferred_host_probe_interval = "5m"

[auth]
session_ttl = "24h"

[[host]]
url = "https://cf.ai-pixel.online/"
priority = 1

[[host]]
url = "https://ai-pixel.online/"
priority = 2

[[host]]
url = "https://api.ai-pixel.online/"
priority = 3

[[account]]
id = "primary"
name = "主账户"
email_env = "user1"
password_env = "password1"
enabled = true
```

账户 `id` 是稳定标识，创建后不得复用于其他账户。`name` 用于页面展示。环境变量名可以沿用现有的 `user1` 和 `password1`，新增账户使用新的环境变量键。

网页登录凭据使用固定环境变量键 `DASHBOARD_USERNAME` 和 `DASHBOARD_PASSWORD`。启动时按以下顺序处理：

1. 直接读取项目 `.env` 中的固定键；网页登录凭据不从 `config.toml` 或进程环境变量覆盖。
2. 如果用户名或密码为空，分别使用密码学安全随机数生成缺失项。用户名使用不易猜测的随机标识，密码至少具有 128 位随机熵。
3. 新生成的值原子写回 `.env`，保留文件中已有的上游账户配置和注释；Linux 上将文件权限限制为 `0600`。
4. 如果需要生成凭据但 `.env` 无法创建、无法安全更新或无法设置安全权限，服务拒绝启动。日志只提示凭据已保存的文件路径，不打印生成值。
5. 数据库初始化后，将 `.env` 中的用户名与密码同步为 SQLite 中的管理员记录；密码使用 Argon2id、独立随机盐和版本化参数进行哈希。
6. 如果 `.env` 凭据发生变化，启动时更新 SQLite 管理员记录并撤销全部现有 Web 会话。

登录请求只与 SQLite 中的带盐哈希比较，不在每次登录时读取或比较 `.env` 明文。`.env` 是启动时的凭据来源，SQLite 是运行期间的认证数据来源。

节点按 `priority` 升序排列；优先级相同时保持 TOML 声明顺序。配置校验失败时服务拒绝启动，并返回不包含凭据值的明确错误。

## 5. 上游协议

当前上游协议基址为 `/api/v1`，使用 Bearer Token。已确认的接口包括：

- `GET /settings/public`：读取登录协议版本等公开设置。
- `POST /auth/login`：提交邮箱、密码和当前登录协议版本。
- `POST /auth/refresh`：刷新访问令牌。
- `GET /usage`：分页读取请求记录。
- `GET /usage/stats`：读取请求统计。
- `GET /usage/balance-ledger`：分页读取余额流水。
- `GET /usage/balance-ledger/stats`：读取余额流水统计。

登录时若上游启用登录协议，客户端读取并提交 `login_agreement_revision`。该行为已经获得项目所有者授权。

访问令牌和刷新令牌只保存在内存中，不写入 SQLite 或日志。令牌到期时优先刷新；刷新失败后重新登录。

## 6. 数据模型

### 6.1 `accounts`

保存账户稳定 ID、显示名、启用状态、当前节点、最近成功同步时间、最近错误摘要和创建/更新时间。不保存密码或上游令牌。

### 6.2 `usage_records`

保存上游请求记录的分析字段，包括：

- 上游记录 ID、账户 ID、请求 ID、API Key ID、上游账户 ID。
- 模型、服务等级、推理强度、入口端点、上游端点、请求类型和流式类型。
- 输入、输出、缓存创建和缓存读取 Token。
- 原始成本、实际成本、余额扣减、点数扣减及费率倍数。
- 总延迟、首 Token 延迟、计费模式、钱包类型和创建时间。
- 未映射的完整原始 JSON。

唯一键为 `(account_id, upstream_id)`。

### 6.3 `balance_ledger_entries`

保存：

- 上游流水 ID、账户 ID、方向、金额、余额、创建时间。
- 原始流水类型 `reason`、业务分类、引用类型和引用 ID。
- 标准化备注文本、可检索备注文本及从备注中提取的结构化字段。
- 完整 `metadata` JSON 和完整原始 JSON。

唯一键为 `(account_id, upstream_id)`。

### 6.4 运行状态表

- `sync_cursors`：每账户、每数据类型的同步游标和最后覆盖区间。
- `sync_runs`：每轮同步的起止时间、状态、读取数量、写入数量和脱敏错误。
- `upstream_health`：节点最近成功/失败、连续失败次数和下次探测时间。
- `dashboard_users`：管理员用户名、Argon2id 参数、随机盐、密码哈希和更新时间；不保存明文密码。
- `web_sessions`：会话令牌哈希、创建时间、过期时间和撤销时间。
- `schema_migrations`：数据库迁移版本。

SQLite 启用 WAL、foreign keys 和 busy timeout。所有迁移在事务中执行。

## 7. 同步流程

### 7.1 初始化同步

新账户或无游标账户从分析时区的上月第一天开始，同步至当前时间。请求记录和余额流水都执行完整分页读取。

### 7.2 增量同步

每分钟为所有启用账户安排一次同步。账户任务可受控并发执行；同一账户同一时刻最多有一轮同步。

每轮同步从上次成功游标减去配置的重叠窗口开始，按创建时间倒序分页读取。记录使用唯一键 UPSERT，因此重叠窗口不会产生重复数据。只有完整分页区间成功后才推进游标。

如果服务长时间停止，恢复后从持久化游标继续补齐，不只读取最近一分钟。

### 7.3 节点切换

首次登录按节点优先顺序尝试。成功后保留当前节点，网络错误、超时或服务端错误触发有限重试和备用节点切换。认证失败不会盲目切换所有节点。

运行在备用节点时，按配置周期探测更高优先级节点；恢复后切回。所有请求具有超时、指数退避和抖动，禁止无限重试。

### 7.4 数据兼容

已知字段解析为结构化列，完整上游 JSON 同时保留。未知字段不会导致整轮同步失败。缺少上游 ID 或创建时间等核心字段的记录将被跳过，并写入不包含凭据和令牌的诊断信息。

## 8. 余额备注与流水类型分析

上游备注由 `reason`、`ref_type`、`ref_id` 和 `metadata` 共同表达，不假设存在单一备注字段。

系统保存原始 metadata，并通过确定性规则生成标准化备注和业务分类。规则提取请求 ID、API Key、账户、群组、订阅周期、小时费率等可用字段。

分析维度包括：

- 原始流水类型 `reason`。
- 业务分类，例如用量扣费、兑换码、账户共享收入、退款和人工调整。
- 入账/出账方向。
- 备注关键词。
- 引用类型和引用 ID。
- 账户和时间范围。

未知流水类型或备注结构归入“其他”，但页面仍显示原始类型、原始 metadata 和原始 JSON。系统不调用外部 AI 服务进行分类。

## 9. 聚合与对比

上游时间解析后以 UTC 保存。聚合时转换为 `analysis.timezone`，默认 `Asia/Shanghai`。

默认按小时聚合，可切换为按日。请求指标包括：

- 请求数。
- 输入、输出、缓存创建和缓存读取 Token。
- 原始成本、实际成本、余额扣减和点数扣减。
- 平均总延迟和平均首 Token 延迟。
- 按账户、API Key、模型、入口端点、上游端点、请求类型、流式类型和计费方式拆分。

余额指标包括：

- 入账次数和入账金额。
- 出账次数和出账金额。
- 净变化。
- 按原始流水类型、业务分类和备注类别拆分。

每个时间桶关联前一天同一时间桶和前七天同一时间桶，分别计算绝对差与百分比。比较基数为零时百分比返回空值，并由页面显示为不可计算，避免无穷大或误导值。

## 10. Web API

API 使用 `/api` 前缀，至少包括：

- `GET /health/live`
- `GET /health/ready`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/session`
- `GET /api/overview`
- `GET /api/usage/trend`
- `GET /api/usage/breakdown`
- `GET /api/usage/records`
- `GET /api/ledger/trend`
- `GET /api/ledger/breakdown`
- `GET /api/ledger/entries`
- `GET /api/accounts/status`
- `POST /api/accounts/{id}/sync`

列表接口使用基于页码的分页，并返回总数。分析接口接受账户、时间范围、粒度及对应维度筛选。所有分析和状态接口都要求有效管理会话。

存活与就绪检查不返回账户、同步或数据库内容，可供 Nginx 和进程管理器探测；其他接口均遵循上述认证要求。

## 11. 前端设计

### 11.1 登录页

管理员账号密码登录。登录成功后创建 24 小时服务端会话，支持主动退出。错误提示不区分用户名不存在与密码错误。

### 11.2 总览页

展示请求数、Token、实际成本、入账、出账和净变化 KPI。默认小时趋势，同时提供昨天同期和上周同期两个可独立开关的对比序列。

### 11.3 请求分析页

支持账户、API Key、模型、入口端点、上游端点、请求类型、流式类型、计费方式和时间筛选。展示成本、Token、延迟、缓存命中、排行和明细。

### 11.4 余额流水页

支持账户、方向、原始流水类型、业务分类、备注关键词、引用类型、引用 ID 和时间筛选。入账与出账始终分开展示。

备注分析展示备注类别趋势、次数、金额贡献和高频类别，可下钻到原始流水、标准化备注、结构化字段及原始 metadata。

### 11.5 账户状态页

展示当前节点、最近成功同步时间、数据新鲜度、错误摘要和手动重试。部分账户失败时，其他账户数据保持可用，页面明确标记汇总结果不完整。

页面采用桌面优先的响应式布局，并提供移动端可用视图。加载、空数据、部分失败和完全失败都有独立状态。

## 12. 分析页面认证与安全

- 管理用户名和明文密码仅在启动时从 `DASHBOARD_USERNAME` 和 `DASHBOARD_PASSWORD` 读取；`config.toml` 不包含这些值或对应键名。
- 缺失的用户名或密码由服务生成并安全回写 `.env`，不会输出到日志。
- SQLite 只保存 Argon2id 带盐密码哈希；每个密码使用独立随机盐，哈希格式包含算法版本和参数，便于后续安全升级。
- 登录时从 SQLite 查询管理员并执行恒定时间的哈希验证；运行期间不依赖 `.env` 明文进行密码比较。
- `.env` 凭据变化后，下一次启动同步新的带盐哈希并撤销旧会话。
- 登录成功生成至少 256 位随机令牌；浏览器持有原始令牌，数据库只存令牌哈希。
- 会话有效期固定为配置的 24 小时；登录后轮换会话，退出时服务端撤销。
- Cookie 设置 `HttpOnly`、`SameSite=Lax`、限定路径；生产环境设置 `Secure`。
- 写接口校验 Origin，并使用 CSRF 防护。
- 登录接口按来源地址和用户名组合限速。
- Go 仅信任本机 Nginx 的代理头；不得无条件信任公网传入的 `X-Forwarded-*`。
- 日志统一脱敏，不输出密码、令牌、Cookie 或完整上游响应。

## 13. Nginx 集成

部署文档提供完整 Nginx 示例，至少包含：

- HTTP 跳转 HTTPS。
- TLS 终止与证书配置占位说明。
- 反向代理到 `config.toml` 指定的 Go 地址和端口。
- `Host`、`X-Real-IP`、`X-Forwarded-For` 和 `X-Forwarded-Proto`。
- 请求体大小、代理连接/读取超时和安全响应头。
- 只允许 Nginx 访问 Go 监听端口的建议。

Go 不直接管理公网证书或对外开放 TLS 端口。

## 14. 错误处理与可观测性

- 配置、数据库迁移或监听失败属于启动错误，进程以非零状态退出。
- 单账户同步失败不会中止其他账户任务。
- 上游认证失败、限流、超时、网络错误、服务端错误和格式错误分别分类。
- 每轮同步记录持续时间、读写数量和最终状态。
- 健康检查区分进程存活与数据库可用；账户数据新鲜度通过受保护状态 API 提供。
- 收到关闭信号时停止接受新任务，等待在途事务完成后关闭数据库。

## 17. v1 验收标准

1. 至少两个模拟账户可同时初始化和增量同步；真实配置可增加多个账户。
2. 初始化覆盖上月第一天至当前时间，重启后从游标继续。
3. 每分钟任务不会为同一账户并发重入，重复读取不会产生重复行。
4. 全账户汇总等于各单账户结果之和；单账户筛选不会混入其他账户。
5. 默认小时聚合正确显示昨天同期和上周同期。
6. 余额入账、出账、净额分别正确，流水类型和备注均可筛选与聚合。
7. 未登录用户无法访问分析 API 和页面；登录会话在 24 小时后失效。
8. `.env` 缺少网页登录用户名或密码时，启动流程安全生成缺失项并以 `0600` 权限原子回写；SQLite 只保存带独立随机盐的 Argon2id 哈希。
9. 部分账户失败时其他账户仍更新，页面显示数据不完整和对应账户状态。
10. Go 监听地址和端口完全由 TOML 控制，并可由 Nginx 正常反向代理。
11. Linux 产物是单个可执行文件，运行时不要求安装 Node.js、SQLite 命令或其他应用依赖。

## 18. 设计约束与后续演进

v1 为单实例设计。数据表和查询均以账户稳定 ID 隔离，为后续增加账户管理 UI、数据保留策略、更多管理员和外部数据库迁移保留边界，但不在 v1 提前实现这些能力。
